Make syntax


# Compile
%.o : %.c
	cc -c $<

# Link
% : %.o
	cc -o $@ $<

Produce syntax


# Compile
[%{name}.o]
dep.c = %{name}.c
recipe = cc -c %{c}

# Link
[%{name}]
dep.o = %{name}.o
recipe = cc -o %{target} %{o}
A slight increase in verbosity, but an increase all the same. What does it buy us?

Make: only first prerequisite named

  • DRY violation

Produce: name prerequisites

You don’t have to name them

Make: a single wildcard

  • also: $$ everywhere

Produce: any number of wildcards…

…or even regular expressions…

…or matching conditions.

Make: arcane string manipulation

Produce: just write Python

Make: counterintuitive declarations

Produce: rule attributes

  • with special meaning: target, cond, deps, dep.*, type
  • type is file (default) or task
  • without special meaning: values assigned by matching or explicitly

Design goal: low barrier for users

  • simple, intuitive syntax
  • one file, just drop it in $PATH
After all, I’m competing with Make here which has the enormous advantage of coming with every Unix-like system.

Design goal: low development cost

Should not take too much of my time…

Why Python?

  • no compilation needed
  • rich standard library
    • argparse for command-line options
    • configparser and shlex for parsing Producefiles
    • subprocess for executing recipes
    • logging for info and debugging
